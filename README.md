# IMadeIt

A showcase platform for makers — woodworkers, software developers, illustrators, ceramicists, and anyone else who builds things. Each post pairs an image-led cover with a long-form Markdown write-up of the work behind it: the materials, the process, the decisions, the lessons.

The interface borrows from two different worlds: an Instagram-style grid for browsing visual work at a glance, and a Medium-style detail view for the story behind each piece.

---

## Status

Early stage. Core flows (auth, post creation, browsing, category filtering) are functional. View tracking, comments, and post editing are next on the roadmap.

---

## Features

- **Magic-link sign-in.** No passwords. New users get a profile row provisioned automatically by a database trigger on first sign-in.
- **Post creation.** Cover image upload, Markdown body, and inline images that can be uploaded mid-write and inserted at the cursor position as Markdown image syntax.
- **Curated categories.** A controlled taxonomy of 18 maker-relevant categories (Woodworking, Software, Ceramics, Photography, etc.) — locked at the database level so the tag list stays clean.
- **Public feeds.** A grid of the latest published work, plus per-category feeds (`/t/woodworking`, etc.) and per-user portfolio pages (`/u/[username]`).
- **Profile editing.** Username, display name, and bio.
- **Post detail view.** Full-bleed cover hero with a typeset long-form body rendering Markdown including inline images, lists, code, headings, and GFM tables.

## Architecture notes

A few decisions worth surfacing for anyone reading the code:

- **Authorization lives in the database, not the app.** Every public table has Row Level Security enabled, with policies that constrain reads to published rows and writes to the row's owner (`auth.uid() = author_id`). The Next.js layer doesn't need to remember "is this user allowed to do X" — Postgres refuses unauthorized writes regardless of how they reach it.

- **View tracking is daily-deduplicated and forgery-resistant.** Clients can never insert into `post_views` directly (no policy permits it). Instead they call a `SECURITY DEFINER` Postgres function `record_post_view(post_id, viewer_hash)` which inserts with `ON CONFLICT DO NOTHING` against a `(post_id, viewer_hash, viewed_on)` primary key. A trigger on the resulting insert increments a denormalized `posts.view_count`.

- **Tags are admin-curated, not user-generated.** The `tags_insert_authed` policy was deliberately revoked; only the seed script (running with the service role) can add new tags. Users select from existing chips on the post form, and the form just sends slugs — unknown slugs are silently ignored server-side.

- **Storage uploads are gated by path.** The `post-media` bucket policy permits authenticated inserts only when the object key starts with the user's UID — `${auth.uid()}/...`. The browser uploads directly to Supabase Storage with the user's JWT; no proxy server, no service role on the client.

- **Schema is owned by Drizzle.** `src/lib/db/schema.ts` is the source of truth. SQL migrations are generated by `drizzle-kit generate` and applied by an idempotent runner in `scripts/migrate.mjs` that tracks applied files in a `_migrations` ledger table. RLS policies and triggers ride alongside in hand-written follow-up migrations.

## Local development

### Prerequisites

- Node 20+
- A free Supabase project (https://supabase.com)

### Setup

```bash
git clone https://github.com/AdeelHL/imadeit.git
cd imadeit
npm install
```

Create `.env.local` with credentials from your Supabase project (Settings → API and Settings → Database → Connection string):

```
NEXT_PUBLIC_SUPABASE_URL=https://<project-ref>.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=sb_publishable_...
SUPABASE_SERVICE_ROLE_KEY=sb_secret_...
DATABASE_URL=postgresql://postgres.<ref>:<password>@<region>.pooler.supabase.com:6543/postgres
```

> Note: special characters in the database password (`!`, `&`, `^`, `$`, etc.) must be URL-encoded.

Apply migrations and seed the curated tag list:

```bash
npm run db:migrate
npm run db:seed
```

Run the dev server:

```bash
npm run dev
```

Visit http://localhost:3000.

### Available scripts

| Script                | Purpose                                                |
| --------------------- | ------------------------------------------------------ |
| `npm run dev`         | Start the Next.js dev server with Turbopack            |
| `npm run build`       | Production build                                       |
| `npm run lint`        | ESLint                                                 |
| `npm run db:generate` | Generate a new SQL migration from `schema.ts` diff     |
| `npm run db:migrate`  | Apply pending migrations                               |
| `npm run db:seed`     | Seed/refresh the curated tag taxonomy                  |
| `npm run db:studio`   | Open Drizzle Studio (visual DB browser)                |

## Project layout

```
src/
  app/
    page.tsx                    # public feed (latest posts)
    login/                      # magic-link sign-in flow
    auth/callback/              # OAuth/OTP callback handler
    new/                        # post creation form + server action
    settings/                   # authenticated profile editor
    t/[tag]/                    # category-filtered feed
    u/[username]/               # user portfolio page
    u/[username]/[slug]/        # post detail page
  components/                   # Header, PostCard, Markdown, TagChips
  lib/
    db/
      schema.ts                 # Drizzle schema (single source of truth)
      client.ts                 # postgres-js + Drizzle instance
    supabase/
      server.ts                 # createClient() for Server Components
      client.ts                 # createClient() for Client Components
      middleware.ts             # session refresh helper
    slug.ts                     # slugify + parseTags
    storage.ts                  # construct public URLs for post-media
  proxy.ts                      # Next.js proxy: refreshes Supabase session

drizzle/                        # generated SQL migrations
scripts/                        # migrate runner, tag seeder
```
